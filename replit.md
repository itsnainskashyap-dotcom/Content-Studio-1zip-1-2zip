# Overview

Content Studio AI (Seedance 2.0) is an AI-powered video content creation platform that streamlines video pre-production. It offers AI-driven tools for story generation, video prompt creation, music briefs, and voiceovers, enabling rapid prototyping of video concepts. The platform integrates an Express API for AI services with a React + Vite frontend, aiming to empower creators with efficient content generation.

# User Preferences

The user prefers an iterative development approach. They want to be able to stop and restart AI generation processes. They appreciate clear error messages that guide them to actionable solutions. They prefer a dark editorial design with specific color schemes.

**Technical Implementations & Feature Specifications:**

**API Server:**
- **AI Integrations**: Express API routes for story generation, video prompts, music briefs, and voiceovers using Anthropic Claude (`claude-sonnet-4-6`, `claude-haiku-4-5`).
- **LLM Handling**: Streaming heartbeat for long LLM calls, retries (`MAX_ATTEMPTS = 2`), and dynamic `max_tokens`.
- **Input Validation**: Zod schemas for request body validation.
- **Story Scaling**: Story content scales based on `partsCount`.

**Content Studio AI Frontend:**
- **Project State Management**: Server-backed via PostgreSQL (jsonb `data` column), with in-memory caching.
- **Background Generation**: `<GenerationProvider>` manages AI generation jobs.
- **Story Builder**: Chat-like interface for story and video prompt generation.
- **Video Prompt Generation**: Generates parts with auto-advancement, client-side capping, progress bar, model-aware shot-count budgeting, character continuity, and `copyablePrompt` with expansion/trimming. Supports `normal` (text) and `json` output.
- **Authentication**: Email + password with bcrypt hashing and httpOnly `cs_session` cookie (30-day TTL).
- **Data Management**: JSON export/import and "Copy ALL N parts" functionality.
- **Parallel Generation**: Fixed concurrency of 4 for batched prompt generation.
- **Character Reference Sheets**: Automatically generates 3-panel composite images via Google Gemini, stored in Replit Object Storage.
- **Auto-Frame Generation with QC**: Renders starting/ending stills with Gemini Vision QC for adherence scoring and regeneration.
- **Custom Character Regeneration**: Allows users to regenerate character images with custom prompts.
- **Per-Project Target Video Model**: Selector for 8 video generators (e.g., Seedance 2.0, Veo 3, Sora) each with a hand-tuned dialect and realism character. Model picker is available inline on the prompt panel.
- **Model-Aware Prompt Templating**: AI system prompts use `{{TARGET_MODEL}}` and other placeholders substituted at request time via `applyModelTokens(text, profile)`.
- **Model-Adaptive Scene Cutting**: Adapts total runtime based on the chosen model's max clip length.
- **Project Aspect Ratio**: Selector for aspect ratios (e.g., 16:9, 9:16) influencing prompt generation and frame rendering.
- **Cinema Image Studio (`/cinema`)**: Advanced cinematography page for single hero stills via Magnific Nano Banana Pro (default `nano-banana-pro-flash` = Gemini 3.1 Flash Image, switchable to `nano-banana-pro` = Gemini 3 Pro Image via `MAGNIFIC_IMAGE_MODEL`). Offers pro-style controls (cameras, lenses), built-in/custom recipes, reference images, generation sliders, negative prompt, and an AI Director scoring panel. Nano Banana Pro is the single image-generation backend across the entire app.
- **AI Video Studio (`/video-studio`)**: End-to-end continuous video generator that stitches Story Builder output into an MP4. Uses Cont Pro (photoreal cinematic via Magnific Veo 3.1 image-to-video + Nano Banana Pro opening frames) and Cont Ultra (long-form via Magnific Seedance image-to-video, also with Nano Banana Pro opening frames). Visual storyboard, per-part JSON prompt, and continuity summary written by Claude Sonnet 4.6. Backend exposes `POST /api/video-studio/jobs`, `GET /api/video-studio/jobs` (slim card list for the Library), `GET /api/video-studio/jobs/active`, `GET /api/video-studio/jobs/:id`, `POST /api/video-studio/jobs/:id/cancel`, and `DELETE /api/video-studio/jobs/:id` (cascades chunks + best-effort wipes Object Storage assets). The engine is fire-and-forget in-process; status tracked in DB with CAS-style atomic transitions. The page also persists `activeJobId` to `localStorage` (key `cs:video-studio:activeJobId`) for instant resume. Visual-bible stage (`buildVisualBible`) emits per-asset progress (`Designing character X of N: name…`) via an `onStageMessage` callback, every NB Pro reference call is bounded by a 90 s per-attempt timeout in `nanoBananaAdapter.ts`, and a second `onPartialBible` callback streams the partial visual bible into `videoJobs.visualBible` after every successful NB Pro call so the frontend can render character cards/location/opening frame as they arrive. NB Pro character/location/opening-frame failures now throw fail-loud (no half-built bible reaches the chunk loop, fixing the "characters spinner stuck while scenes start" race). The snapshot returned by `toStatusSnapshot` includes a slim `visualBible` projection (id/name/objectPath only — no b64) consumed by the `VisualBibleGallery` and `ChunkTile` (per-scene video preview + last-frame poster) components on the page for a live playground experience.
- **Magnific (Freepik) migration (April 2026)**: All image + video generation routes through a single `FREEPIK_API_KEY` against the Magnific REST API (`https://api.magnific.com/v1/ai`) via the shared helper `artifacts/api-server/src/video/providers/magnificClient.ts` (sends both `x-magnific-api-key` and the legacy `x-freepik-api-key` header for compatibility). The previous direct `@google/genai` paths for Veo and image generation are gone. Slugs in use: `text-to-image/nano-banana-pro` and `nano-banana-pro-flash` for stills, `image-to-video/veo-3-1` (Cont Pro high) and `veo-3-1-fast` (Cont Pro standard), and the existing Seedance slug for Cont Ultra. The `image` field accepts inline base64, so we never need a public starting-frame URL. `GOOGLE_GENAI_API_KEY` is now optional and only used by the Image Studio QC vision route (`checkGeminiQC` in preflight); boot does not depend on it.
- **Cont Pro chaining**: Part 1 of every Cont Pro job uses the visual bible's opening frame as the `image` (image-to-video). Parts 2..N use the captured last frame of the previous chunk as the `image`, pinning continuity. Cont Ultra (Seedance I2V) follows the same opening-frame pattern. Veo I2V on Magnific does not accept reference images, so the prior `referenceImages` plumbing for Cont Pro has been removed; the opening frame plus the chained last frame carry the cast/location lock. (Reference-to-Video would require public URLs we don't host, so it's intentionally out of scope.) Seedance is verified image-to-video ONLY — Magnific exposes no `video-to-video/seedance-*` slug and the I2V endpoint accepts a single `image` field (no first+last-frame, no `reference_video_url`), so the chained-last-frame approach is the maximum continuity the API allows.
- **Pixel-level visual lock for opening frames (April 2026)**: `nanoBananaAdapter` accepts a `referenceImages: Array<{b64,mimeType}>` arg and forwards it as `reference_images` (Magnific NB Pro hard-cap = 3) so generation, not just the prompt, is anchored to actual locked pixels. `storyboardEngine.buildVisualBible` calls a new `collectOpeningReferences(characters, locations)` helper that picks the primary location plate + up to 2 story-order character plates and feeds them to BOTH the primary and simplified-fallback opening-frame renders. The same set is mirrored into `qc.references` so Gemini vision QC scores the opening against the actual locked images instead of a paraphrase. The inline opening-recovery path in `runContinuousLoop` does the same thing using the saved visual bible. Characters and the primary location are intentionally generated text-only — feeding character A as a ref while making character B blends faces in NB Pro, so cross-character chaining is by design avoided. Net effect: Part 1's image-to-video is now seeded by an opening still that pixel-matches the cast and world the rest of the chunks then propagate via last-frame chaining.
- **Chunk bridge frames (April 2026)**: Seedance / Veo are image-to-video and only see the SINGLE starting frame, so when the captured last frame of part N-1 lands on a moment with no character on screen (wide landscape, object cutaway, empty room), part N would lose all pixel-level memory of the cast and the face would visibly drift. `storyboardEngine.buildChunkBridgeFrame` runs a fresh NB Pro re-render at every chunk N>=2 with refs = `[previousLastFrame, characterRefs...]` — slot 1 always preserves world/lighting/grade/composition from chunk N-1's last frame, slots 2-3 re-assert locked faces from the visual bible. **Beat-aware character selection**: the helper scans the chunk's `visualStoryboard.startingFramePrompt` + `storyContext.currentBeat` for character-name mentions; characters NOT named in the upcoming shot are excluded from refs AND the spec switches to `no_characters_in_frame:true` with a `["people","human figures"]` negative — so an intentional cutaway / b-roll / establishing wide is rendered character-free instead of having the cast force-planted. The bridge spec uses the chunk's `visualStoryboard.startingFramePrompt` so the rendered frame matches both the upcoming beat AND the prior visual state; the result becomes the actual `image` handed to Seedance / Veo. QC runs lenient (threshold 6, 1 retry) since the bridge is an internal stepping stone, not a deliverable. Best-effort: any NB Pro failure logs and falls back to the raw previous last frame so a bridge hiccup never kills a long job. Emergency disable via `VIDEO_STUDIO_DISABLE_CHUNK_BRIDGE=1`. Cost: +1 NB Pro call per chunk past part 1.
- **JSON prompt resilience**: `JsonVideoPromptSchema` (`claude46Adapter.ts`) uses field-level `.catch()` so a single malformed Claude field never breaks the parse; `engineModel` is normalized case-insensitively in `transform`.
- **Pipeline-fix invariants (April 2026)**: Every prompt sent to NB Pro (image), Veo 3.1 (Cont Pro) and Seedance (Cont Ultra) is a JSON-formatted string capped at **4500 chars including spaces**, enforced by `lib/promptEnvelope.ts → buildJsonPrompt(spec, opts)`. Phase 1 drops least-critical top-level keys per `dropOrder`; phase 2 iteratively shaves the longest deeply-nested string in the spec tree (never coerces objects to `[object Object]…`); phase 3 last-resort emits a minimal valid-JSON subset. Visual-bible character/location/opening prompts all flow through this envelope. Chunk-loop prompts are serialized via `serializeVideoJsonPrompt` (the legacy `jsonPromptToNaturalLanguage` flattener was removed). `lib/imageQcValidator.ts → validateImageWithGemini({...,strict})` runs Gemini 2.5 Flash vision over every NB Pro image; in-pipeline calls use `strict:true` so validator outages count as fail and trigger up to 2 QC retries that re-serialize the spec with an injected `avoid_issues:[...]` field (preserving the JSON invariant — no plain-text concatenation onto JSON). After retries are exhausted the loop ships the last image with its qc verdict attached. Cont Ultra (Seedance) request body now sends `generate_audio:true` AND a structured `audio_config` block alongside Cont Pro; both adapters accept a `BuiltAudioSpec` produced by `lib/hinglishPhonetics.ts → buildAudioSpec(direction, language, "veo"|"seedance")`, which applies Devanagari→Latin syllable breaks and a "Ready." prepend for Hinglish voiceovers (English voiceovers untouched). Self-test: `pnpm --filter @workspace/api-server exec tsx scripts/promptEnvelopeSelfTest.mts` (14 invariants).
- **30-day asset library**: `videoJobs.expiresAt` (default `now() + 30 days`) drives `sweepExpiredJobs` invoked from `sweepStaleVideoJobs` on every API boot — purges row, cascades chunks, and best-effort deletes Object Storage assets via `ObjectStorageService.tryDeleteObjectEntity`. The Library section in `history.tsx` renders a card grid with thumbnail/opening frame, character thumbs, days-remaining badge, download MP4, open-in-studio, and confirm-delete (refuses delete on still-running jobs with HTTP 409). `deleteJob(id, ownerId?)` accepts an optional owner id; the manual-delete route always passes `req.user!.id` so the destructive SQL is gated on `WHERE id = ? AND owner_id = ?` (no TOCTOU window vs the route's ownership read). The TTL sweeper passes no ownerId because rows are dropped on TTL grounds independent of ownership.

# Deployment Architecture

- **Target**: Autoscale on Replit, custom domain `contentstudioai.in`.
- **Routing**: Replit's artifact-router (`router = "application"` in `.replit`).
  - `/` and all SPA routes → `contentstudio-ai` artifact, served as native static (`serve = "static"`, publicDir `artifacts/contentstudio-ai/dist/public`, SPA rewrite `/* → /index.html`). No Node process needed for the React app in production.
  - `/api/*` → `api-server` artifact (Node process on local port 8080, mapped to external 80 via `[[ports]]`).
- **Builds**: Each artifact's `[services.production.build]` runs at deploy time. The `.replit` `[deployment.build]` pre-hook also builds the frontend (redundant but harmless).
- **Critical lesson**: An artifact that claims `paths` but has no `[services.production]` block causes Cloud Run to return HTTP 500 (`Server: Google Frontend`, body `Internal Server Error`) for every matching path. Either give the artifact a production block (process or static) or remove its production path claim.

# External Dependencies

- **Replit AI Integrations**: Anthropic Claude (story / prompt / director).
- **Magnific (Freepik) API**: Single `FREEPIK_API_KEY` powers ALL image + video generation — Nano Banana Pro for images, Veo 3.1 I2V for Cont Pro, Seedance for Cont Ultra.
- **Google Gemini (optional)**: Only used by the Image Studio QC vision scoring route via `GOOGLE_GENAI_API_KEY`. Boot does not depend on it; missing the key only fails the QC route at call time.
- **Replit Object Storage**: Private bucket for generated/uploaded image bytes, with 302 redirect to signed GCS URLs. Includes a local-disk fallback for infrastructure outages.
- **PostgreSQL**: Database for `users`, `sessions`, and `projects`.
- **bcrypt**: Server-side password hashing.
- **Zod**: Schema validation.
- **Drizzle ORM**: ORM for PostgreSQL.
